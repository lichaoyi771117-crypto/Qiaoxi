Grill-Me 审计：Qiaoxi 项目设计方案 v1.0
我先逐条梳理设计方案中的决策树，对每个关键节点进行追问和验证。

审计问题 1：State 0 画像校准的两轮结构
设计方案中写的是"6道封闭式点选题"，但没有区分战略层（0-B）和战术层（0-C）的两轮结构。

PRD 原文：

0-B：战略层画像（Q1-Q3），乔曦清淡模态
0-C：战术层画像（Q4-Q6），乔曦锋锐模态
设计方案中遗漏了两轮之间的模态切换（清淡→锋锐），这是一个关键遗漏——因为模态切换是乔曦输出风格的硬约束。

我的推荐修改：在 State 0 描述中明确标注 0-B（清淡模态）→ 0-C（锋锐模态），并说明两轮之间乔曦的声明语切换。

审计问题 2：PAKTON 改造深度是否匹配实际代码？
设计方案说 PAKTON Interrogator 改造为"6实例并行 + 记忆隔离"。但实际 PAKTON 的 Interrogator 是基于 LangGraph StateGraph 的单实例多步推理循环（generate_question → get_answer → refine_answer → router 循环），不支持多实例并行。

改造不仅仅是"复制6份"这么简单。需要：

编写并行调度器（ThreadPoolExecutor 或 asyncio）
每实例分配独立 thread_id 保证记忆隔离
每个实例注入不同的 system prompt（六人各自角色）
超时控制和异常容器化（一个实例崩溃不影响其余5个）
设计方案中对此描述过于笼统。需要补充并行调度架构的具体实现方案。

审计问题 3：RAG 法规数据质量——条款级切分能做到吗？
设计方案承诺"按条款级切分（'第X条'为分割标记）"。

Chinese-Dataset-Laws 确实做到了每条独立成行，但 lawtext-laws 的 Markdown 文件内部结构不统一——有的用 ## 第X章 分级，有的直接列条款。需要对 2477 个文件逐一验证切分质量，否则 RAG 检索结果会出现"切到半条"或"一整章全部返回"的噪音。

设计方案未提及切分质量验证方案。应补充：随机抽样 100 个文件做切分质量人工验证，切分准确率需 ≥95%。

审计问题 4：推演引擎 70% 置信度承诺的底层逻辑
设计方案中将综合置信度预估为 ~75%，加权方式是以主观确定性百分比简单加权。但这不是"置信度"的正确定义——统计学置信度需要基于样本量和方差计算。

在 MVP 阶段，推演引擎的输出更应被定性为"方向性指示"而非"概率预测"。设计方案中的置信度数字（资金流 ~90%、政策风险 ~50%）可能会给用户造成过度量化错觉。

推荐修正：将置信度改为"确定性评级"（高/中/低），避免使用百分比数字。给出 70% 的方向性置信承诺而非统计置信承诺。

审计问题 5：六君子的 Prompt 设计质量
设计方案只说了"注入 ai-legal-claude 的 10 维风险评分体系"，但六人各自的思维工具箱（资金安全边际测算表、不对称性检测清单、行业尽调缺口分析、资金-权利映射表、运营控制权检查点、对手盘恶意推演、杠铃策略建议）需要自建 Prompt 模板。

这 7 个工具的设计质量决定了六君子输出的专业度。设计方案没有说明每个工具的具体结构和输出格式。

补充建议：每个工具的 Prompt 模板至少需包含（1）输入变量（2）计算/推理逻辑（3）输出格式（4）一票否决触发条件。

审计问题 6：合同模板库"积木化"的 MVP 边界
PRD §7.3 定义了合同模板库的三维索引（行业 × 交易类型 × 风险等级）和 5 个核心积木模块。State 8 的重构方案包依赖此模板库。

第3个问题：RAG 法规数据质量控制

设计方案目前对法规数据源有 2 个（lawtext-laws 和 Chinese-Dataset-Laws），但我需要指出一个事实：这两个数据集的原始数据都来自国家法律法规数据库（flk.npc.gov.cn）——这是全国人大维护的官方权威来源。因此数据权威性不存在问题，但存在 3 个需要你决策的隐患：

隐患 A：时效性

lawtext-laws：标注"共有 723 件，其中有效 347 件"，说明包含了已废止的法律
Chinese-Dataset-Laws：公司法同时包含 2018 版和 2023 版
风险：如果 ChromaDB 检索同时返回新旧版本，LLM 可能引用已废止法条
隐患 B：覆盖缺口

缺少地方性法规（PRD §7.2 要求"项目所在省份地方性法规"）
缺少最新的司法解释（2024-2026 年期间新增的）
隐患 C：条款切分精度

从 Markdown 到 ChromaDB 的切分规则如果设计不好，会出现"切断法条"或"embedding 粒度过粗"的问题
我的建议方案：

时效性：在导入 ChromaDB 时，对每条法条标注 生效状态 元数据字段；在检索时，代码层过滤 已废止 标识的法条；同一法律多版本并存时，仅保留最新版进入活跃索引
覆盖缺口：MVP 阶段先接受"国家级法律法规完整，地方性法规空缺"的状态，在报告中标明"法规检索范围限于国家级法律法规"；后续引入省级法规作为增量更新
条款切分：使用"第X条"作为分割锚点，每条独立成 chunk；增加上下文窗口（前后各保留 1 条）防止切断后语义不完整
你是否接受这个方案？还是希望 MVP 阶段就解决地方性法规问题？

我的回复：1.我们的法律法规库，需要删除所有已经废止的法律，理由：我们是本地小型RAG，没必要留存失效的法律文档；2.我们专注于合同审查，重点收录国家重要的法律法规，至于地方性行政法规，不强制要求，没有也没关系；3.条款切分精度属于纯粹技术问题，你负责规划解决。
