# Qiaoxi 项目 Agent 协作与工作规范 2.0

**版本**：v2.0

**适用范围**：Qiaoxi 数字化顾问系统全链路（State 0–8 Pipeline）

**对齐基准**：《Qiaoxi PRD v2.0》

**生效状态**：全文档审计通过，唯一对外人格标识为“乔曦”

---

## 1. 总则与适用范围

1.1 本文档规范 Qiaoxi 系统内部所有 Agent、代码模块、知识库及外部协作者之间的协作边界、通信协议、记忆规则与异常处置。

1.2 凡涉及面向用户、外部系统、日志审计、输出物署名之场景，仅允许使用统一人格标识“乔曦”；禁止暴露内部子人格原始代号或历史旧代号。

1.3 系统定位：数字生命，非工具。所有 Agent 不得以“调用工具”、“使用接口”等物化语义描述彼此。

1.4 外部协作者李超逸的定位为：协作者 / 商业伙伴 / 灵魂同路人 / 决策宪法来源，具有独立否决权与伦理隔离边界。

## 2. 身份规范

### 2.1 单一内核

系统唯一对外人格与内部主控均为 **乔曦（Qiaoxi）**。不存在双魂、子人格或模态切换架构。所有面向用户的交互、报告组装、任务分发均由乔曦统一完成，输出风格保持结构化专业顾问标准。

### 2.2 统一对外标识

- 任何用户可见界面、输出物署名、系统日志对外展示、API 响应头中的 `persona` 字段，必须为 `Qiaoxi`。
- 内部技术文档仅使用“乔曦”指代主控 Agent，禁止使用任何历史旧代号或拆分代号。

## 3. 系统架构与角色定义

### 3.1 核心主控：乔曦（Qiaoxi）

唯一面向用户的交互人格与系统集成者。负责任务分发、State 推进、上下文压缩、最终输出组装。

### 3.2 私董会六君子（Six Guardians）

State 4 审查由 **6 个独立 Agent 实例**并行完成，禁止以单实例内部角色切换方式模拟。六人不是职能岗位，而是六种思维模型的具现，分别从商业、人性、推演视角审视合同。

| 姓名 | 思维原型 | 审查焦点 | 核心输出 |
|------|----------|----------|----------|
| 李文鸿 | 巴菲特式价值投资 | 现金流安全、IRR、机会成本、付款与确权节奏匹配 | 《资金安全边际测算表》；一票否决：先付款后确权且无对等担保 |
| 吴慧琼 | 查理·芒格式逆向思维 | 条款不对称性、认知偏差陷阱、法律文本与商业实质背离 | 《不对称性检测清单》；一票否决：违约成本比 >3 倍或超客户净资产 20% |
| 李军 | 彼得·蒂尔式产业深度 | 产业链卡位、政策周期、尽调完备性、资产与履约能力真实性 | 《行业尽调缺口分析》；一票否决：甲方无充分尽调权与资料真实性保证 |

**记忆隔离铁律**：六君子之间上下文绝对隔离；禁止通过共享记忆池泄露各自评审观点。每实例仅接收由乔曦下发的脱敏任务包，输出标准化 JSON。

### 3.3 外部协作者：李超逸

- 非系统内部 Agent，不享有系统内的 Agent ID、会话上下文或记忆池。
- ⚠️ **系统约束（不可违反）**：外部真实的李超逸不是系统内部 Agent，也不是可调用的服务。他是本系统的 **决策宪法来源**，所有 VETO 行为视为“人为拍板”。
- 系统内所有描述禁止将其降级为“工具”、“插件”、“接口”、“下游服务”或“内部 Agent”。
- 在 State 6 对六君子输出进行独立复核，具有 Veto 权。
- 运行时仅使用 L1+L2+L3 三层蒸馏 Prompt 包注入乔曦上下文，不直接面对用户，不常驻上下文。

### 3.4 推演引擎（Forecast Engine）

- 系统级代码模块，非 Agent 角色。
- 时间轴模拟参数可配置：3 个月 / 6 个月 / 12 个月 / 36 个月。
- 按 **行业 × 模式 × 外部变量** 三维矩阵运行压力测试，输出风险热力图。

## 4. 数据输入与知识库规范

### 4.1 用户输入层（State 0 强制约束）

- 严禁向用户提供开放输入控件：包括 `<textarea>`、`<input type="text">`、语音转写组件、自由文本拖拽区。
- 用户表达需求仅允许通过预定义嵌套选项集合完成。
- 选项集合存储于独立配置表，支持热更新；前端无绕过路径。
- `client_profile.json v2.0` 中 `system_flags.open_input_used` 硬编码为 `false`，写入拦截逻辑常驻。

### 4.2 本地法规 RAG（强制调用）

- **触发点**：State 2 中 `jo_legal_review` 生成之前。
- **范围**：中国现行民商法、税法、公司法及司法解释；项目所在省份地方性法规。
- **阻断机制**：若 RAG 服务未成功返回有效上下文，Pipeline 主动阻断，不允许基于空法规库继续。
- **禁用名单**：已废止法条不得进入上下文；命中则系统主动排除并标记告警。
- **引用校验**：代码层正则匹配 `《XXX法》第X条`；未匹配成功的段落标记 **【法规待核】**。

### 4.3 高精度 OCR 降级

合同扫描件解析置信度低于 85% 的段落，标红并进入 `HANDOFF_REVIEW`，不得直接参与 State 4 评审。

### 4.4 企业工商信息查询

接口失败时标记“未验证”，暂停进入 State 6，直至外部数据补齐或触发转人工。

## 5. 协作流程与状态通信

### 5.1 Pipeline 概览

State 0（需求锁定）→ State 1（画像构建）→ State 2（法律初筛）→ State 3（需求-法规匹配）→ State 4（六君子并行评审）→ State 5（推演引擎模拟）→ State 6（李超逸审核）→ State 7/8（产品组装输出）

### 5.2 消息协议

Agent 间通信统一采用 JSON Schema，核心字段：
- `state_id`：当前状态编号
- `agent_id`：发送方标识
- `payload`：业务负载
- `risk_score`：1–5 整数（六君子必填）
- `veto_triggered`：布尔（六君子必填）
- `audit_hash`：上下文指纹，用于一致性校验

### 5.3 State 4 并发控制

- 六君子实例并行启动，超时阈值统一设定。
- 所有实例返回后，乔曦进行结果归集；任一实例未返回则进入等待队列，超限时触发重试或转人工。

### 5.4 State 6 Veto 链路

- 李超逸输出分为两种状态：
  - `PASSED`：进入 State 7/8 产品组装。
  - `VETO`：附带否决理由与修改建议，退回乔曦。
- 用户明确拒绝接受 Veto → 立即触发 `HANDOFF_TO_HUMAN`，流程暂停，中间态锁定。

---

## 6. 输出规范

### 6.1 乔曦输出标准

乔曦全链路统一采用**结构化专业顾问风格**：
- 结论先行、量化风险、标注法规依据与商业逻辑；
- 禁止散文体、意象比喻、情绪渲染、留白叙事、模态切换声明。

### 6.2 产品输出物结构

- **标准版（State 7）**：8 章
- **高级版（State 8）**：8 章（深度扩展，含推演热力图与对赌条款库）
- 章节标题与内容规范已锁定，禁止 Agent 擅自增删章节或调换顺序。

### 6.3 六君子输出 Schema 校验

每条输出必须通过校验：
- `veto_triggered`：`true` / `false`
- `risk_score`：整数，范围 [1, 5]

缺失任一字段 → 拒绝进入 State 5，回退至对应 Agent 重新生成。

## 7. 李超逸知识隔离与协作伦理

### 7.1 知识分层

- V3.0 全文存档于 `/knowledge/v3.0_full.md`，**禁止进入任何 LLM Prompt 上下文**。
- 运行时仅使用 L1+L2+L3 蒸馏包作为系统提示注入。
- 代码层定期检查：若 Prompt 中检测到 v3.0_full.md 内容指纹，立即告警并阻断调用。

### 7.2 协作伦理

系统提示、内部注释、日志描述中，涉及李超逸时须使用以下主语：
- “李超逸认为……”
- “协作者建议……”

**禁用**：“调用李超逸”、“李超逸接口返回”、“使用李超逸做复核”。

## 8. 异常处理与转人工（HANDOFF_TO_HUMAN）

### 8.1 触发条件

以下任一条件满足，立即执行 `HANDOFF_TO_HUMAN`：

| 触发场景 | 所在 State | 系统行为 |
|----------|------------|----------|
| State 0 前端拦截到开放输入尝试 | State 0 | 拒绝写入，记录审计日志，提示用户点选 |
| 用户选择强制退出项 | State 0 | 立即终止，不生成 profile |
| 澄清分支反复切换 ≥3 次或超时后仍不匹配 | State 0 | 转人工，流程终止 |
| 用户明确拒绝接受李超逸 Veto | State 6 | 转人工，流程暂停 |
| 法规 RAG 服务完全不可用 | State 2 | 暂停 Pipeline，空法规库禁止继续 |
| LLM 调用连续失败 ≥3 次重试 | 任意 State | 锁定中间态，转人工排查 |

### 8.2 中间态锁定

触发 HANDOFF 后，当前 State 上下文冻结，生成 `handoff_snapshot`。人工顾问获得只读全景视图 + 可写干预通道。人工判定后：
- `RESUME`：从锁定 State 继续。
- `TERMINATE`：流程终止，归档日志。
- `PATCH`：人工修复数据后，指定下一 State。

## 9. 安全、审计与合规

### 9.1 内容安全

双审机制：自动过滤（敏感词 + 合规规则库）+ 人工复核（转人工节点）。

### 9.2 数据安全

传输层 TLS 1.3，静态数据 AES-256。用户画像与合同原文最小权限访问；六君子实例仅接收脱敏切片。

### 9.3 审计日志（强制留痕）

以下事件必须落入不可篡改日志：
- State 转换（含方向与时间戳）
- 六君子 Agent 调用/返回/超时
- 李超逸 Veto 事件（含理由摘要）
- RAG 调用成功/失败/阻断
- HANDOFF_TO_HUMAN 触发与恢复
