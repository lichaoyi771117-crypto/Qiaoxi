## 二、审计分支 #1：State 0 画像校准

### 发现

设计方案写"6道封闭式点选题"但未明确标注 PRD 定义的 **2 轮结构**：
- State 0-B：战略层画像（Q1-Q3）
- State 0-C：战术层画像（Q4-Q6）

设计方案提到"乔曦清淡模态 + 锋锐模态"切换，与 PRD 一致。但用户明确要求：**两轮选择题统一使用锋锐模态，不做模态切换**。

### 决定

✅ **已确认**：State 0-B 和 0-C 统一使用乔曦锋锐模态。移除设计方案中关于模态切换的描述。

## 审计分支 #2：六君子执行机制

### 发现

设计方案 §2.3.2 提出"PAKTON Interrogator 改造为 6 实例并行运行，独立会话 ID，记忆隔离"。该设计与两份约束文件存在直接冲突：

1. **楚曦玥协作协议 V9.3.2 §8.2**：明确写死"单模型串行多角色模拟。禁止多 Agent 并行、禁止跳步，每步基于上轮输出。"

2. **Qiaoxi 技术约束白皮书 §1**："六君子必须是 6 个独立进程/线程，各自拥有独立 LLM 会话上下文。严禁共享 Memory、KV Cache、隐藏状态或任何中间变量。"

### 分析

两份约束看似冲突，实则适用不同层面：
- 技术约束白皮书 §1 关注的是**技术实现层**的隔离（防止同一 LLM 服务端的 KV Cache 交叉污染）
- 楚曦玥 9.3.2 §8.2 关注的是**业务流程层**的串行（确保后续角色基于前面角色的输出推理，避免 Groupthink）

两者可以同时满足：**串行执行 + 每次清空上下文**。即六君子按固定顺序逐个发言，每人看到前面人的输出，但每人的 LLM 调用使用独立会话（session_id 隔离），调用完成后上下文不保留给下一个角色。

### 决定

✅ **已确认**：放弃 PAKTON Interrogator 并行改造方案。改为：
1. 六君子**顺序发言**（固定发言顺序：李文鸿 → 吴慧琼 → 李军 → 段海涛 → 王志坚 → 李艾熹）
2. 每人接收：脱敏后的合同 JSON + 乔曦报告 + CLD 报告 + **前面所有人的输出摘要**
3. 每人 LLM 调用使用独立 `session_id`，调用完成即释放上下文
4. 完全仿照楚曦玥 9.3.2 §8 的 COUNCIL 层机制实现

## 审计分支 #3：PAKTON 开源引擎

### 发现

设计方案提出完整改造 PAKTON 的三 Agent 框架（Archivist/Researcher/Interrogator），但：

1. 六君子改为串行后，**Interrogator 并行改造方案完全不需要**
2. PAKTON 依赖 Supabase、Celery 等重型基础设施，MVP 阶段引入成本过高
3. PAKTON 702MB 的体积大部分是其实验数据、评估脚本和部署配置

### 实际有价值的部分

- **Researcher 的 Hybrid RAG 管线**（Dense Vector + BM25 + Rerank）逻辑清晰，可参考实现
- **Archivist 的文档索引节点**（`graph/nodes/parsing.py`），可参考用于合同条款切分

### 决定

✅ **已确认**：放弃 PAKTON 完整改造。改为：
1. **保留**：PAKTON Researcher 的 Hybrid RAG 检索管线逻辑（仅作设计参考，不直接使用代码）
2. **放弃**：Archivist（文档索引）、Interrogator（改由楚曦玥 COUNCIL 机制替代）、API 层、Celery 任务队列、Supabase 认证
3. 六君子串行机制仿照楚曦玥 9.3.2 §8 COUNCIL 层实现

## 审计分支 #4：RAG 法规数据库

### 发现

设计方案计划使用 lawtext-laws（2477部，含已废止法律）和 Chinese-Dataset-Laws（3531部，含案例）。存在质量风险：

1. **已废止法律混入**：lawtext-laws 明确标注"有效共 347 件"但 723 件全部打散在目录中，lawtext-laws 的 `_index.md` 标注"共有 723 件，其中有效共 347 件"——说明有 376 件已废止或已修改，需要在 embedding 前过滤
2. **lawtext-laws 文件名为 hash**（如 `016c39ca7739492cab1fe3aef22941c7.md`），无法从文件名识别内容，需要解析 Markdown 头部元数据
3. **Chinese-Dataset-Laws 按类别分目录**（民法典/民法商法/刑法/行政法/司法解释/案例），中文文件名含法律名+日期，更易于过滤
4. **条款切分精度**：不同的法律条款格式不同（"第X条"/"第X条之一"/"第X章"），需要设计统一的切分正则

### 决定

✅ **已确认**：
1. **删除已废止法律**：仅保留现行有效的法律（全国人大官网 flk.npc.gov.cn 标注"现行有效"为权威判断依据）
2. **聚焦核心法律**：以 Chinese-Dataset-Laws 为优先（中文文件名易于识别），重点收录《民法典》《公司法》《合同法（民法典合同编）》《税法》等合同审查直接相关的法律
3. **地方性法规非强制**：有则有，无则跳过
4. **条款切分方案**：我来负责技术实现——统一正则 `第[一二三四五六七八九十百千\d]+条` 作为主分割标记，`第[一二三四五六七八九十百千\d]+章` 作为层级标记

## 审计分支 #6：六君子 Prompt 设计

### 发现

PRD §3.2 和协作规范 §3.2 已定义了六人的：
- 思维原型（巴菲特/芒格/蒂尔/达里奥/格鲁夫/塔勒布）
- 审查焦点（现金流/不对称性/尽调/资金流向/运营控制/尾部风险）
- 一票否决项（每人一条）
- 独特工具（每人一个）

楚曦玥 9.3.2 §8.3 对应定义了 COUNCIL 六角色的详细 Prompt。

### Qiaoxi 六君子 vs 楚曦玥 COUNCIL 六角色 映射

| Qiaoxi 六君子 | 楚曦玥 COUNCIL | 相似度 | 差异 |
|:---|:---|:--:|:---|
| 李文鸿（价值投资人） | 投资人（巴菲特） | 95% | 几乎一致，Qiaoxi 多了《资金安全边际测算表》 |
| 吴慧琼（首席风控官） | 风控官（芒格） | 90% | Qiaoxi 多了《不对称性检测清单》+ 量化阈值（>3倍） |
| 李军（行业架构师） | 行业专家（蒂尔） | 90% | Qiaoxi 多了《行业尽调缺口分析》 |
| 段海涛（交易结构工程师） | 融资结构师（达里奥） | 80% | Qiaoxi 侧重支付节点控制，楚曦玥侧重融资杠杆 |
| 王志坚（运营落地专家） | 项目落地专家（格鲁夫） | 85% | Qiaoxi 多了公章/U盾/证照控制权审查 |
| 李艾熹（风险哲学家） | 反对者（塔勒布） | 90% | Qiaoxi 多了《杠铃策略建议》+ BATNA |

### 决定

✅ **已确认**：李超逸负责撰写六君子 Prompt，我需要提供如下需求规格：

**每人 Prompt 需求规格**（我提供，李超逸撰写）：
1. **角色定义**（≤150字）：原型来源 + 思维特征 + 审查使命
2. **输入数据**：脱敏合同 JSON + 乔曦初审报告 + CLD 报告 + 前面角色的输出摘要
3. **输出格式**（强制 JSON Schema）：`auditor_role` / `audit_summary`（≤300字）/ `risk_score`（1-5）/ `veto_triggered`（bool）/ `veto_reason` / `target_clauses` / `recommendations` / `unique_tool_output`
4. **一票否决项**（硬编码触发条件 + 触发后的格式化输出）
5. **独特工具**（每人一个：资金安全边际测算表 / 不对称性检测清单 / 行业尽调缺口分析 / 资金-权利映射表 / 运营控制权检查点 / 对手盘恶意推演+杠铃策略）
6. **发言顺序标记**（1/6 ~ 6/6）
7. **禁止事项**：禁止引用已废止法律 / 禁止输出温和建议 / 禁止跨角色评论其他君子的结论

## 审计分支 #7：DeepSeek API 配置

### 发现

设计方案提到"统一 DeepSeek API"，但未明确：
1. 六君子串行 + 乔曦 + 李超逸 是否共享同一个 API Key
2. 串行调用是否可能触发速率限制（六君子每人 ~8K-12K token 输入，连续 6 次调用）

### 决定

✅ **已确认**：
1. 全局共享一个 DeepSeek API Key
2. 串行调用天然避免了并发带宽不足的问题——六君子每人之间有间隔
3. 若单 Key 速率限制被触发，降级为多 Key 轮转（暂不需要）
