## 摘要

Qiaoxi（乔曦）是程信霖咨询为商业咨询顾问打造的一款 AI 驱动的商业合同审查与决策重构系统。区别于传统合同审查工具以律师视角输出法律意见书的模式，Qiaoxi 从商业咨询顾问视角出发，输出《商业决策报告》——直接告诉客户"能不能签、怎么改、谈不拢怎么退"。

项目以 **Python + Streamlit** 为技术底座，以 **DeepSeek API** 为推理引擎，以 **55,088条中国法律法规** 为本地 RAG 知识库，构建了一条覆盖"画像校准→合同解析→法律审查→商业模式解构→六人审计→推演辩论→决策输出→合同重构"的完整 9-State 管线。经 6 类合同实测，系统在全流程稳定性、报告质量、PII 安全和重构能力四个维度均达到交付标准。

本文从开发目标、方法论基础、技术架构、各模块实现、测试验证、关键问题解决、交付物清单、局限性与展望八个方面，对项目进行全面总结。

## 一、项目开发目标

### 1.1 业务目标

程信霖咨询在商业咨询实践中发现：客户拿到一份商业合同后，最核心的诉求不是"这份合同的法律风险有哪些"，而是**"我该不该签？如果要改，怎么改？如果谈不拢，什么条件下该退出？"**。传统法律审查工具和法律意见书无法回答这些问题，因为它们缺失三个维度：

1. **客户画像**：不知道客户是谁、怕什么、底线在哪
2. **商业建模**：不知道合同交易结构的系统性风险
3. **决策推演**：不知道不同选择在未来不同时间窗口的后果

Qiaoxi 的建设目标就是填补这三个空白，把合同审查从"法律风险清单"升级为**"商业决策报告"**。

### 1.2 技术目标

| 目标 | 说明 |
|------|------|
| **全本地 RAG** | 法规检索零外部 API 依赖，使用清洗后的中国法律法规数据库（1,132部法 / 55,088条） |
| **强制 RAG 触发** | State 2 法律初审未调用 RAG 则管线阻断，禁止 LLM 自行编造法条 |
| **多 Agent 并行审计** | 6 个独立 LLM 会话，记忆物理隔离，一票否决权，串行固定顺序 |
| **确定性推演** | 纯 Python 确定性函数执行风险推演，禁止 LLM 生成推演结论 |
| **多层决策蒸馏** | L1 硬编码规则（100% 确定性）→ L2 人格核 → L3 领域规则，逐步注入 |
| **零开放输入** | State 0 画像采集阶段禁止自由文本输入，仅封闭式点选（单选/多选/下拉） |
| **全链路安全** | PII 脱敏（8种信息类型）+ AES-256 加密 + WORM 审计日志 |
| **合同重构** | State 8 基于审查结果自动生成新合同草案、框架协议和尽调清单 |

## 二、核心方法论

### 2.1 商业咨询五步法

Qiaoxi 的方法论基础是程信霖咨询独创的"商业咨询五步法"：

```
Step 1: 客户画像与利益校准         → State 0
Step 2: 商业模式系统动力学解构     → State 3
Step 3: 核心风险识别（法律+商业+人性）→ State 2 + State 4
Step 4: 后果推演与不对称性评估     → State 5
Step 5: 杠杆解重构                 → State 6 + State 8
```

五步法确保审查结论始终锚定在客户的具体利益上，而非抽象的法律条文中。

## 四、各模块实现详解

### 4.1 State 0：客户画像校准（`src/state0_profile.py`，179 行）

**核心约束**：零开放输入（`textarea { display: none !important; }` CSS 强制执行），仅封闭式点选。

**实现方式**：
- **第一轮**（R1）3 道题：合同身份识别 + 核心利益权重 + 最大风险感知。题目由 LLM 根据脱敏后的合同内容动态生成
- **第二轮**（R2）3 道追问：风险偏好 + 最大损失容忍度 + 谈判地位 + BATNA + 可妥协维度 + 绝对底线。题目由 LLM 根据 R1 回答动态生成
- **兜底机制**：LLM 调用失败时使用硬编码题目（6道固定选择题）
- **输出**：`client_profile.json`（strategic_layer + tactical_layer + system_flags）

**设计理念**：用户不能说"我担心合同有风险"这种模糊的话，必须通过选择来精确化自己的立场。

### 4.2 State 1：合同解析（`src/state1_parse.py`，207 行）

**核心功能**：将 PDF/DOCX 合同文件解析为结构化条款树。

**三级降级解析策略**：

```
DOCX → python-docx 直接提取（精度最高，无编码问题）
PDF  → pdfplumber（x_tolerance=3, y_tolerance=3，正确处理 Identity-H 中文字体）
PDF  → PyMuPDF / fitz 兜底
```

**条款树提取**：正则匹配 `第[一二三四五六七八九十百千\d]+条` 作为分割标记，按序编号为 `CLS-0001`、`CLS-0002`……等内部 ID。同时保留 37 类风险关键词的正则预标定。

**关键突破**：Identity-H 中文编码乱码问题。部分中文 PDF 使用 Identity-H 编码的字体，PyMuPDF 直接提取会乱码。通过引入 pdfplumber 作为第二级解析器，有效解决。

### 4.3 State 2：乔曦法律初审（`src/state2_legal_review.py`，173 行）

**核心约束**：强制触发本地法规 RAG，未调用 RAG 则管线阻断。

**工作原理**：
1. 接收 `clause_tree`（State 1）+ RAG 检索结果（`rag_retriever.py`）+ 客户画像（State 0）
2. 从客户立场出发，逐条分析每个条款
3. 每项风险标注：有利/不利/中性 + risk_level + 法律依据 + 建议动作
4. 与客户绝对底线冲突的条款自动标为 high risk
5. RAG 未命中时标注 `【法规待核】`，禁止 LLM 自行编造法条

**输出格式**（`jo_legal_review.json`）：
```json
{
  "client_position_summary": "基于客户立场的审查角度概述",
  "risks": [{
    "clause_id": "CLS-0004",
    "risk_level": "high|medium|low",
    "client_impact": "对客户有利|不利|中性",
    "description": "风险描述",
    "legal_basis": "《法规名称》第X条（现行有效）",
    "suggested_action": "修改建议"
  }],
  "bottom_line_violations": ["底线冲突条款列表"],
  "overall_client_assessment": "综合评估"
}
```

### 4.4 法规 RAG 引擎（`src/rag_loader.py` 161 行 + `src/rag_retriever.py` 137 行）

**数据规模**：

| 指标 | 数值 |
|------|------|
| 法律部数 | 1,132 部（仅现行有效） |
| 条款条数 | 55,088 条 |
| 数据来源 | Chinese-Dataset-Laws（3,531 部原始）+ lawtext-laws（2,477 部补充） |
| 清洗后大小 | 24.5 MB（JSON格式） |
| 优先类别 | 民法典、商法、经济法、行政法、司法解释等 57 个核心关键词 |

**检索原理**：
1. 从合同文本提取法律关键词（~60 个中文法律术语映射表）
2. 在清洗后的 330K 行 JSON 中进行关键词匹配
3. 精确匹配优先，模糊匹配兜底
4. 按相关度排序，取 top-K（默认 10 vector + 3 BM25）

**为什么不用 ChromaDB**：MVP 阶段优先简化依赖。bge-m3 模型（~2GB）+ ChromaDB 的向量检索管线已在 `rag_loader.py` 中预留，后续可平滑升级。

### 4.5 State 3：商业模式解构（`src/state3_cld.py`，82 行）

**核心功能**：从合同条款中提取商业模式的系统动力学结构。

**提取三个维度**：
1. **资金流向**：谁付钱、付给谁、多少钱量级、什么时候付
2. **权力分配**：股权/公章/决策权/法人在谁手里、什么时候转移
3. **时间约束**：签约→付款→交割→到期之间的先后依赖关系

**输出因果回路图**（CLD）：
- **增强回路（R）**："滚雪球效应"——正向放大，一旦启动难以停止
- **调节回路（B）**："刹车机制"——负反馈制约，自动纠正偏差
- 至少 2 个回路（1R + 1B），至多 5 个

**容错设计**：如果合同文本信息不足，对应字段标注"信息不足"，不强行推断。

### 4.6 State 4：六位评审员串行审计（`src/state4_council.py`，424 行）

这是系统中架构设计最复杂的模块。

**六位评审员定义**：

| 序号 | 角色 Key | 姓名 | 思维原型 | 审查焦点 | VETO 条件 |
|------|---------|------|---------|---------|-----------|
| 1/6 | `value_investor` | 李文鸿 | 巴菲特 | 资本保全、安全边际 | 先付款后确权且无对等担保 |
| 2/6 | `cfo_risk` | 吴慧琼 | 芒格 | 不对称性、认知偏差 | 违约成本比 > 3x 或击穿净资产 20% |
| 3/6 | `industry_architect` | 李军 | 蒂尔 | 产业链卡位、尽调完备性 | 甲方无充分尽调权与资料真实性保证 |
| 4/6 | `deal_engineer` | 段海涛 | 达里奥 | 资金-权利映射、退出路径 | 大额资金无共管/托管/担保 |
| 5/6 | `operations` | 王志坚 | 格鲁夫 | 运营控制权、交割可实现性 | 公章/执照/U盾由对手单方持有 |
| 6/6 | `risk_philosopher` | 李艾熹 | 塔勒布 | 尾部风险、对手盘恶意推演 | 尾部风险毁灭性且不可逆 |

**关键约束**：
- **串行固定顺序**：1/6 → 2/6 → … → 6/6，不可并行
- **记忆隔离**：每人独立 LLM 调用，不共享 KV Cache
- **VETO 不中断执行**：任一人否决后继续执行完全部 6 人
- **6/6 李艾熹是反对派**：唯一被授权攻击前序五人共识的角色
- **输出校验**：每份审计结果经 JSON Schema 校验（veto 须有理由，risk_score 须在 1–5）
- **失败重试**：最多重试 1 次，仍失败则输出兜底（risk_score=3，建议人工复核）

**每位评审员的输出结构**：auditor_role + speaking_order + audit_summary（≤400字） + risk_score（1–5） + veto_triggered + veto_reason + target_clauses + recommendations + unique_tool_output（角色专属工具输出）

### 4.7 State 5：推演引擎 + 辩论合成（`src/state5_simulation.py`，235 行）

**两部分组成**：

**SimulationEngine（确定性推演）**：
- **纯 Python 确定性函数**，禁止 LLM 生成推演结论（设计铁律）
- 在 4 个时间切片（3/6/12/36 个月）上模拟风险演变
- 增强回路（R）随时间放大风险，调节回路（B）随时间抑制风险
- 合成信号：早期以 B 回路为主（缓冲期），晚期以 R 回路为主（惯性期）
- VETO 加权：任一人否决，所有切片风险等级上调一级
- 输出轨迹标签：逐渐恶化/稳定向好/风险稳定/波动

**DebateSynthesizer（LLM 辩论合成）**：
- 交叉比对 6 位评审员意见
- 标记共识（≥4 人共同关注的 clause_id）与少数观点（1–2 人关注）
- 共识分析是确定性函数（`_analyze_consensus()`）
- 生成 ≤500 字的《私董会研讨纪要》

### 4.8 State 6：李超逸决策引擎（`src/state6_decision.py`，205 行）

**三层蒸馏包设计**：

| 层级 | 类型 | 大小 | 内容 |
|------|------|------|------|
| **L1** | 硬编码 if-then | 6条规则 | 六戒律：资本安全垫 / 显失公平 / 信息不对称 / 运营失控 / 尾部风险不可逆 / 资金失控 |
| **L2** | LLM 注入 | <800 tokens | 李超逸人格核：四选一决策风格、结论先行、3 句依据原则 |
| **L3** | LLM 注入 | <1000 tokens | 并购专用规则：付款-确权顺序、公章与 U 盾、税务敞口、法人变更、竞业禁止、退出机制 |

**关键设计**：
- L1 硬编码优先检查，触发即**直接 VETO**，不调 LLM
- L1 未触发才将六位评审员摘要 + 推演简报 + 客户画像简报 + L2 + L3 注入 LLM
- 输出四选一标签：**签（Sign）/ 改（Modify）/ 拖（Defer）/ 退（Abandon）**
- 若 LLM 选择忽略 L1 判决（override VETO），同样触发 HANDOFF_TO_HUMAN
- 李超逸常驻禁止：L1+L2+L3 仅在 State 6 注入，防止污染乔曦（State 2）的输出

### 4.9 State 7：标准版报告生成（`app.py` 内嵌）

**报告结构**（8 章）：

```
一、合同概览      — 合同名称、条款计数、交易结构摘要
二、您的核心关切   — 画像映射 + 底线冲突检测
三、交易结构分析   — CLD 通俗解读 + 资金流向 + 权力分配 + 时间线 + 推演
四、法律风险分析   — 高/中/低三级法条分析 + 法律依据 + 行动建议
五、商业风险透视   — 六位评审员意见汇总（角色名过滤为视角标签）
六、决策建议       — 签/改/拖/退 + 3 句决策理由
七、免责声明
```

**双格式下载**：Markdown（`.md`） + Word（`.docx`，自研 Markdown→DOCX 转换器，微软雅黑字体，支持标题/加粗/表格/列表/引用块/水平线）。

### 4.10 State 8：合同重构引擎（`src/state8_reconstruct.py`，340 行）

**付费高级服务**（¥30）：

**四步串行生成**：
1. **Step 1 — 深度消化**（max_tokens=2048, json_object）：生成内部"重构任务书"
2. **Step 2 — 新合同草案**（max_tokens=6000）：逐条重构，附【重构说明】注释
3. **Step 3 — 防御性框架协议**（max_tokens=4096）：仅并购/股权转让/采矿权场景
4. **Step 4 — 尽调清单**（max_tokens=6000）：六大类核查，仅并购/采矿权场景

**智能判定 `files_to_generate`**：
- `contract_draft`：始终生成
- `framework_agreement`：默认 false，仅并购/股权转让/采矿权/千万级标的 + 尽调缺失时为 true
- `dd_checklist`：默认 false，仅并购（股权收购/资产收购/合并分立）和采矿权/探矿权转让为 true
- 底线原则："若在斟酌是否该设为 true——就设为 false"

**三档重构方向**：aggressive（最大限度维护我方）/ neutral（市场惯例）/ balanced（保核心 + 降摩擦）

### 4.11 安全模块（`src/security.py`，396 行）

**四层防护**：

| 层级 | 机制 | 说明 |
|------|------|------|
| **脱敏** | 正则 + 关键词匹配 | 8 类 PII：身份证/手机/银行卡/信用代码/邮箱/公司名/人名/地址/金额 |

## 七、关键架构决策

| 决策 | 内容 | 理由 |
|------|------|------|
| **RAG 强制触发** | State 2 未调 RAG 则管线阻断 | 防止 LLM 编造法条（法律领域零容忍） |
| **六位评审员串行非并行** | 固定顺序逐一执行，后续角色仅接收前序摘要 | 防止群体思维（groupthink），保留少数意见 |
| **推演引擎非 AI** | 纯 Python 确定性函数 | 推演结论必须可复现、可审计 |
| **L1 硬编码优先** | 六戒律 if-then 在 LLM 之前执行 | 底线规则必须 100% 确定，不可被 LLM"说服绕过" |
| **李超逸常驻禁止** | L1+L2+L3 仅在 State 6 注入 | 防止前置状态（乔曦/评审员）被决策者人格污染 |
| **State 0 零开放输入** | 禁止 textarea 渲染 | 防止用户输入模糊需求导致后续分析偏差 |
| **DeepSeek 主力 + Sonnet 攻坚** | 分层成本控制策略 | 常规推理用低价模型，复杂攻坚切换高性能模型 |
| **ChromaDB 预留非阻塞** | MVP 使用纯关键词检索 | 快速验证管线逻辑，后续可平滑升级向量检索 |
