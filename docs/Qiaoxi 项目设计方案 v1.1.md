## 一、项目定位与设计原则

### 1.1 一句话定位

律师输出《法律意见书》（告诉你有什么风险）；Qiaoxi 输出《商业决策报告》（直接告诉你能不能签、怎么改、谈不拢怎么退）。

### 1.2 设计原则

| 原则 | 来源 | 说明 |
|------|------|------|
| **咨询视角优先** | 原始需求 V2.1 | 商业咨询五步法，非律师逐条审阅 |
| **1+6+1 团队架构** | PRD v2.0 §3 | 乔曦(主控+法务) + 六君子(串行发言) + 李超逸(最终决策) |
| **六君子串行顺序发言** | 楚曦玥 9.3.2 §8.2 | 单模型串行多角色模拟，禁止并行、禁止跳步，每步基于上轮输出 |
| **六君子独立会话隔离** | 技术约束 §1 | 每人独立 LLM 会话 ID，调用完成即释放上下文，禁止跨角色 KV Cache 共享 |
| **李超逸常驻禁止** | PRD §3.3 | L1+L2+L3 仅在 State 6 注入 |
| **推演引擎非 Agent** | 技术约束 §4 | 纯 Python 确定性计算，禁止 LLM 生成推演结论 |
| **FSM 状态机** | 技术约束 §5 | State 0-8 显式实现，禁用 >3 层嵌套 if-else |
| **本地 RAG 强制** | 技术约束 §2 + PRD §7.2 | ChromaDB + bge-m3 + BM25，禁止外网向量服务 |
| **零开放输入** | PRD §5.1 | State 0 仅封闭式点选，textarea 禁止渲染 |
| **AES-256 加密** | 数据安全铁律 §1 | 绝密/敏感级中间态 JSON 加密存储 |
| **拿来主义** | 原始需求 §5 | 开源引擎 + 自建适配层，不重复造轮子 |

### 2.3 开源引擎选型与改造方案

#### 2.3.1 合同文档解析引擎 —— 基于 PactGuard-ERNIE-PP 改造

**选型理由**：
- MIT 协议，商用友好
- 四阶段工作流（解析→风险分析→建议生成→结果渲染）与 Qiaoxi State 1-2 天然对应
- PP-StructureV3 版面恢复能力对中文合同 PDF 支持好
- MCP Service 独立部署，松耦合

**改造方案**：

| 改造项 | 原实现 | 改造后 | 原因 |
|--------|--------|--------|------|
| **LLM 后端** | ERNIE 4.5（百度） | DeepSeek API（OpenAI 兼容） | 统一 LLM 栈，降低成本 |
| **Prompt 模板** | 通用合同风险识别 | 乔曦专属：中国法规RAG引用 + 法律/商业双维标定 | 适配 Qiaoxi PRD §3.1 |
| **输出 Schema** | 自有 JSON 格式 | `jo_legal_review.json`（PRD §6.4 定义） | 对接 State 2 标准输出 |
| **法规引用** | LLM 自由生成 | 强制触发本地 ChromaDB RAG，代码层正则校验 `《XXX法》第X条` | 技术约束 §2 + PRD §4.1 |
| **MCP Service** | 保留 | 保留，增加合同条款结构化提取 | 产出 `clause_tree.json`（PRD State 1） |
| **paddlepaddle 依赖** | 3.2.1（~2.5GB） | 保留，作为可选 OCR 降级通道 | 扫描件 OCR 置信度 <85% 时触发 |

**改造代码存放**：`D:\Ai RAG\Qiaoxi\modified code\PactGuard-Qiaoxi\`

**改造要点**：
1. `contract_workflow.py` 第 44 行 `default_model = "ernie-4.5-turbo-128k"` → `"deepseek-chat"`
2. `_analyze_risks()` 方法增加 RAG 调用钩子，读取 ChromaDB 检索结果后注入 Prompt
3. 风险 JSON 输出增加 `legal_basis` 字段（法条引用）和 `rag_confidence` 字段（检索置信度）
4. 新增合同条款树（`clause_tree`）结构化提取模块

#### 2.3.2 六君子串行机制 —— 仿楚曦玥 COUNCIL 层自建

**选型理由**：
- 楚曦玥协作协议 V9.3.2 §8.2 明确规定"单模型串行多角色模拟。禁止多 Agent 并行、禁止跳步，每步基于上轮输出。"
- PAKTON Interrogator 的并行改造方案因架构约束变更不再需要
- DeepSeek API 串行调用避免了并发带宽不足问题

**实现方案**：

| 实现项 | 说明 |
|--------|------|
| **发言顺序** | 李文鸿(1/6) → 吴慧琼(2/6) → 李军(3/6) → 段海涛(4/6) → 王志坚(5/6) → 李艾熹(6/6) |
| **每人输入** | 脱敏合同 JSON + 乔曦初审报告 + CLD 报告 + **前面所有人的输出摘要** |
| **会话隔离** | 每人独立 `session_id`，LLM 调用完成后上下文立即释放，不保留给下一个角色 |
| **输出格式** | 强制 JSON Schema：`auditor_role` / `audit_summary`(≤300字) / `risk_score`(1-5) / `veto_triggered`(bool) / `veto_reason` / `target_clauses` / `recommendations` / `unique_tool_output` |
| **第二轮讨论** | 用户显式指令触发，固定顺序：主持人提取分歧(≤3个) → 各角色修正(≤100字) → 反对者优先攻击共识 → 主持人收敛 |
| **参考原型** | 楚曦玥 9.3.2 §8.3 六角色决策闭环 + §8.4 决策流程固定顺序 |

### 2.4 法规数据库 RAG 方案

#### 2.4.1 数据源

| 数据源 | 规模 | 格式 | 说明 | 用途 |
|--------|------|------|------|------|
| **Chinese-Dataset-Laws** | 3531 部 | Markdown，中文文件名（含法律名+日期） | 按类别分目录，易于按名称过滤 | 🥇 主库 |
| **lawtext-laws** | 2477 部 | Markdown，hash 文件名 | 按法律/行政法规/司法解释/监察法规分目录 | 🥈 补充库 |

#### 2.4.2 数据清洗规则

1. **删除已废止法律**：仅保留现行有效的法律。Chinese-Dataset-Laws 文件名含日期标注（如 `公司法(2018-10-26).md` vs `公司法(2023-12-29).md`），可通过解析文件名和 Markdown 头部的修订记录判断有效性；lawtext-laws 的 `_index.md` 标注"有效共 347 件"，可以该元数据为辅助判断依据
2. **聚焦核心法律**：重点收录与合同审查直接相关的法律——民法典（含合同编）、公司法、税法、民商法、担保法、招投标法等
3. **地方性法规非强制**：有则有，无则跳过
4. **合同范本/案例保留**：Chinese-Dataset-Laws 含"案例"目录，保留作辅助参考但不纳入核心法条 RAG

#### 2.4.3 条款切分方案

```
[原始 Markdown 法规文件]
       ↓
按 "第X条" 为分割标记切分（主正则）：第[一二三四五六七八九十百千\d]+条
按 "第X章" 为层级标记（辅助正则）：第[一二三四五六七八九十百千\d]+章
       ↓
每条格式化为："《法律名称》第X条 [条款内容]"
       ↓
bge-m3 Embedding → ChromaDB 向量存储
       ↓
BM25 索引构建（用于精排）
```

**切分精度处理**（技术细节，由开发者负责）：
- "第X条之一/之二" 作为独立条款
- "第X款" 作为父条款的子片段，保留层级关系
- 跨段落条款合并为一个 Chunk

#### 2.4.4 RAG 检索流程（State 2 强制触发）

```
[合同条款文本]
       ↓
bge-m3 向量检索 → ChromaDB Top-10 候选法条
       ↓
BM25 精排 → Top-3 最相关法条
       ↓
代码层正则校验："《XXX法》第X条" 格式
       ↓
注入乔曦 Prompt → LLM 生成审查意见
       ↓
未匹配成功的标注 "【法规待核】"
已废止的法条主动排除并告警（不应到达此步骤，此为兜底）
```

#### 2.4.5 法规数据库约束

- **范围**：中国现行民商法、税法、公司法及司法解释
- **禁用名单**：所有标注"已废止"的法条不得进入上下文；命中则主动排除并告警
- **引用格式**：`《法规名称》第X条第X款（生效状态：现行有效/已废止/待核）`
- **阻断机制**：RAG 服务未成功返回有效上下文 → Pipeline 主动阻断，不允许基于空法规库继续
- **更新方式**：离线脚本替换，不自动拉取，人工审核后替换
- **体积约束**：仅保留现行有效法律，目标库体积控制在 ~50MB 以内

### 2.5 推演引擎方案

#### 2.5.1 设计原则

- **纯 Python 确定性函数**，禁止调用 LLM 生成推演结论（技术约束 §4）
- **MVP 目标**：达到 70% 置信度

#### 2.5.2 推演逻辑

```
[State 3 CLD 报告]
       ↓
提取关键变量：
  - 资金变量：支付总额、支付节点、共管金额、对等担保金额
  - 权力变量：公章持有方、股权比例、董事会席位、法人代表
  - 时间变量：签约日、交割日、付款节点日期、合同到期日
       ↓
设定 4 个时间切片：T+3月 / T+6月 / T+12月 / T+36月
       ↓
每个切片运行：
  1. 资金流模拟：按支付节点推算各方累计投入/回收
  2. 权力转移模拟：确权节点 vs 付款节点的时间差分析
  3. 不对称性指标：甲方违约成本 / 乙方违约成本
  4. 压力测试：政策变化 ±20% / 价格波动 ±30%
       ↓
输出《推演快照》（simulation_snapshot.json）
  - 每个时间切片的关键指标变化
  - 风险热力值（1-5）
  - 不对称性拐点标记
```

#### 2.5.3 推演结果置信度评估

| 推演维度 | 确定性程度 | 说明 |
|---------|:--------:|------|
| 资金流向时间序列 | ~90% | 基于合同明确约定的支付节点，确定性高 |
| 权力转移分析 | ~85% | 基于股权/公章/治理结构条款，较为确定 |
| 违约不对称性 | ~75% | 部分依赖对违约概率的估算，有一定不确定性 |
| 政策风险压力测试 | ~50% | 置信度最低，仅作警示性参考 |
| **综合置信度** | **~75%** | 加权平均，超过 70% 目标 ✅ |

## 三、工作流程与标准

### 3.1 FSM 状态机（State 0-8）

> 严格遵循 PRD v2.0 §4 定义。以下仅标注与开源引擎改造相关的技术实现细节。

```
[State 0: 画像校准与接收]
 ├─ 0-A：合同文件接收（PDF/DOCX/图片）+ 基础信息结构化录入
 ├─ 0-B：战略层画像（Q1-Q3），乔曦锋锐模态
 ├─ 0-C：战术层画像（Q4-Q6），乔曦锋锐模态
 │  ⚠️ 注意：0-B 和 0-C 统一使用锋锐模态，不做模态切换
 ├─ 0-D：参数锁定，生成 client_profile.json（零开放输入）
 ├─ 技术实现：Streamlit 前端，封闭式点选组件（radio/selectbox），禁用 textarea
 ├─ 审计字段：system_flags.open_input_used 硬编码为 false
 └─ 退出路径：选择"以上均不符合"→ HANDOFF_TO_HUMAN

[State 1: 文档解析与结构化]
 ├─ 技术实现：PactGuard 改造版 MCP Service + 自研条款树提取
 ├─ PDF → PP-StructureV3 版面恢复 → 结构化文本
 ├─ 条款结构化：按"第X条"/"第X章"/"Section X" 分割 → clause_tree.json
 ├─ OCR 降级：扫描件置信度 <85% → 标红进入 HANDOFF_REVIEW
 └─ 输出：contract_raw.json + clause_tree.json

[State 2: 乔曦初审 + 法规RAG]
```

## 五、数据安全与合规

### 5.1 数据三级分类

完全继承《数据安全铁律（MVP 版）》§1 的分类标准。

| 级别 | 数据范围 | 处理规则 |
|------|----------|----------|
| **绝密** | 合同原文、client_profile、工商查询原始回包 | 永不离开本地运行环境，不上传任何第三方云 |
| **敏感** | 中间态 JSON、审计日志分析摘要、六君子输出 | AES-256 加密存储，最小权限访问 |
| **公开** | 报告模板、Prompt 模板、系统配置枚举值 | 明文存储，版本控制管理 |

### 5.2 六君子输入脱敏规则

完全继承《数据安全铁律（MVP 版）》§2。

| 实体类型 | 脱敏动作 | 示例 |
|----------|----------|------|
| 自然人姓名 | 替换为 `甲方`/`乙方`/`丙方` | `"张三"` → `"甲方代表"` |
| 公司/机构名 | 替换为 `公司A` / `公司B` | `"霖信莯咨询"` → `"甲方"` |
| 金额数字 | 替换为 `金额X元`，保留量级 | `"5000万元"` → `"金额X元（千万级）"` |
| 证件/账号 | 正则匹配后替换为 `[已脱敏]` | 身份证、手机号、银行账号 |
| 地址/地块 | 替换为 `某省某市` | 精确地址模糊化 |

PII 正则规则库本地维护，匹配失败则**阻断进入 State 4**。

### 5.3 TTL 自动删除

完全继承《数据安全铁律（MVP 版）》§3。

| 数据类型 | 保留期限 | 动作 |
|----------|----------|------|
| 合同原文（PDF/DOCX/图片） | 任务完成/取消后 30 天 | `shred` 级物理删除，不可恢复 |
| 中间态 JSON（绝密/敏感） | 任务完成/取消后 30 天 | 同上 |
| `handoff_snapshot` | 人工恢复后即时删除，或 90 天无操作删除 | 同上 |
| 审计日志 | **永久** | 只读，不可改，不可删（WORM） |

### 5.4 审计日志（强制留痕）

完全继承《数据安全铁律（MVP 版）》§4。以下事件强制落盘 **WORM** 存储：

- State 转换（含时间戳与方向）
- 六君子 Agent 调用/返回/超时（每人独立记录）
- 李超逸 Veto 事件（含理由摘要）
- RAG 调用成功/失败/阻断
- HANDOFF_TO_HUMAN 触发与恢复
- 数据脱敏命中/失败记录

### 5.5 对外安全

- 传输层：本地 Streamlit 服务，MVP 阶段不对外暴露
- 静态数据：AES-256 加密，解密密钥由环境变量注入，不硬编码于仓库
- LLM 调用：仅向 DeepSeek API 发送脱敏后的条款文本（非完整合同原文）；六君子收到的材料已预先 PII 脱敏

## 六、异常处理与转人工

完全继承 PRD v2.0 §8 和协作规范 2.0 §8。

### 6.1 HANDOFF_TO_HUMAN 触发条件

| 场景 | 触发 State | 系统行为 |
|------|------------|----------|
| State 0 开放输入被拦截 | 0 | 拒绝写入，记录审计日志，提示用户点选 |
| 用户选择强制退出项 | 0 | 立即终止，不生成 profile |
| 澄清分支反复切换 ≥3 次或超时后仍不匹配 | 0 | 转人工，流程终止 |
| 用户明确拒绝接受李超逸 VETO | 6 | 转人工，流程暂停 |
| 法规 RAG 服务完全不可用 | 2 | 暂停 Pipeline，空法规库禁止继续 |
| LLM 调用连续失败 ≥3 次重试 | 任意 | 锁定中间态，转人工排查 |
| 任一六君子超时 ≥2 次 | 4 | 记录超时角色，跳过该角色继续或转人工 |

### 6.2 中间态锁定

触发 HANDOFF 后：
- 当前 State 上下文冻结，生成 `handoff_snapshot`
- 人工顾问获得只读全景视图 + 可写干预通道
- 人工指令：`RESUME`（继续）/ `TERMINATE`（终止）/ `PATCH`（修复后指定下一 State）

#### E.3 六人各自需求规格

##### 1/6 李文鸿（价值投资人）

- **原型**：巴菲特式价值投资
- **审查焦点**：客户资金的时间价值、交易总对价的合理性、付款节奏与确权节奏的匹配、沉没成本陷阱
- **独特工具**：《资金安全边际测算表》——输出付款节点与确权节点的对比矩阵，标记"先付款后确权"时间差
- **一票否决**：若交易结构要求"先付款后确权"，且资金一旦离手无对等担保 → `veto_triggered=true`，理由："先付款后确权且无对等担保"
- **Prompt 设计要点**：强调现金流为王、IRR 思维、安全边际概念；要求量化资金时间价值损失

##### 2/6 吴慧琼（首席风控官）

- **原型**：查理·芒格式逆向思维
- **审查焦点**：条款中隐藏的不对称性、认知偏差陷阱、法律文本与商业实质的背离
- **独特工具**：《不对称性检测清单》——逐条计算甲方违约成本 / 乙方违约成本，标记 >3 倍或超过客户净资产 20% 的条款
- **一票否决**：若甲方违约成本 / 乙方违约成本 > 3 倍（或绝对额差异超过客户净资产 20%）→ `veto_triggered=true`，理由："显失公平，必须重构"
- **Prompt 设计要点**：强制使用检查清单法、双轨分析；要求阅读前面角色输出后指出未被发现的认知偏差

##### 3/6 李军（行业架构师）

- **原型**：彼得·蒂尔式产业深度
- **审查焦点**：合同所涉行业的尽调完备性、政策合规门槛、资产真实性、乙方履约能力的可验证性
- **独特工具**：《行业尽调缺口分析》——列出合同应有但未提供的尽调资料清单，标注每项的获取成本与不可获取时的替代方案
- **一票否决**：若合同未赋予甲方充分的尽职调查权与资料真实性保证 → `veto_triggered=true`，理由："信息不对称风险极高"
- **Prompt 设计要点**：基于 `basic_info.industry` 激活行业专项知识；强制检查产业链卡位和政策周期

##### 4/6 段海涛（交易结构工程师）

- **原型**：瑞·达里奥式债务与资金流向
- **审查焦点**：每一笔钱的支付节点与对应权利的获取节点是否一一对应；是否存在资金监管真空；退出路径是否畅通
- **独特工具**：《资金-权利映射表》——以表格形式列出每笔资金支付 vs 对应权利转移，标记时间差与监管缺失
- **一票否决**：单笔超过 1000 万元或总金额 30% 以上的资金支付，若无共管/托管/对等担保机制 → `veto_triggered=true`，理由："资金失控风险"
- **Prompt 设计要点**：强制绘制资金流向图（文本化）；要求检查是否存在资金监管真空期

##### 5/6 王志坚（运营落地专家）

- **原型**：安迪·格鲁夫式执行与偏执
- **审查焦点**：公司治理（公章、证照、U盾、法人代表）、运营禁止条款、人章分离风险、交割先决条件、时间节点可实现性
- **独特工具**：《运营控制权检查点》——逐项检查公章/营业执照/审批U盾/法人代表的持有与托管安排
- **一票否决**：公章/营业执照/审批 U 盾由交易对手单方持有，且无律所/银行托管机制 → `veto_triggered=true`，理由："运营失控风险"
- **Prompt 设计要点**：以 Only the Paranoid Survive 为基调；强制检查交割先决条件的可实现性

##### 6/6 李艾熹（风险哲学家 / 反对派）

- **原型**：纳西姆·塔勒布式反脆弱
- **审查焦点**：如果对手方是恶意设计而非不专业，哪些条款是最致命的暗器；不可承受的最坏情况；合同外的替代方案（BATNA）
- **独特工具**：《对手盘恶意推演》——Assume Bad Faith，逐条推演最坏情境；《杠铃策略建议》——提出 BATNA
- **一票否决**：若最坏情况（尾部风险）对客户是毁灭性且不可逆的 → `veto_triggered=true`，理由："不可接受"
- **Prompt 设计要点**：这是最后发言的角色，必须优先攻击前面 5 人已形成的共识；优先否定不负责建设；检查必要条件缺失/逻辑断裂/AI幻觉/不可验证假设；字数限制可放宽至 ≤400 字

#### E.4 Prompt 通用约束（所有人遵守）

1. **禁止引用已废止法律**：所有法条必须来源于乔曦初审报告中的 RAG 检索结果
2. **禁止输出温和建议**：必须给出明确结论，禁止"建议权衡""视情况而定"等模糊表述
3. **禁止跨角色评论**：不得评论或质疑其他君子的结论（仅李艾熹例外，可攻击前面角色的共识）
4. **禁止对商业模式合理性做主观判断**（留给李超逸）
5. **输出必须为合法 JSON**：严格遵循输出 Schema
6. **一票否决必须给出量化依据**：不触发则 `veto_triggered=false`，`veto_reason` 留空
